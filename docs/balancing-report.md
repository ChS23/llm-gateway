# Сравнение стратегий балансировки

## Стратегии

### Round-Robin

Циклический перебор: `counter % backends.len()`. Per-model AtomicUsize, lock-free.

**Плюсы:** простота, предсказуемость, нулевой overhead.
**Минусы:** не учитывает разницу в латентности или нагрузке провайдеров.
**Когда:** все реплики одинаковы по мощности.

### Weighted Round-Robin

Cumulative weights — провайдер с weight=3 получает 3 из 4 запросов при weight [3,1].

**Плюсы:** учитывает разную мощность.
**Минусы:** статичен, не адаптируется к runtime.
**Когда:** известно соотношение мощностей заранее.

Подтверждено тестами: weights [3,1] → 75%/25% распределение (100 запросов: a=75, b=25).

### Latency-Based

Exponential Moving Average (EMA) per provider в Redis. Decay factor 0.3 — реагирует за ~5 запросов.

```
new_ema = 0.3 × observation + 0.7 × old_ema
```

Sliding window 5 минут для фильтрации старых данных.

**Плюсы:** адаптируется к текущей нагрузке, выбирает самого быстрого.
**Минусы:** нужен Redis, задержка адаптации (~5 запросов), overhead на запись.
**Когда:** провайдеры с переменной латентностью (OpenAI API latency варьируется от 200ms до 5s).

### Least-Connections

`AtomicUsize` per provider — инкремент при начале запроса, декремент при завершении. Выбирает провайдера с минимумом in-flight.

**Плюсы:** учитывает длительность запросов, не только количество.
**Минусы:** не учитывает скорость провайдера.
**Когда:** запросы разной длительности (короткие completions vs длинные reasoning chains).

### Health-Aware

Round-robin с фильтром: unhealthy провайдеры исключаются из пула через circuit breaker.

**Плюсы:** автоматический обход нездоровых, failover.
**Минусы:** overhead на проверку health state.
**Когда:** ненадёжные провайдеры, requirement на высокую доступность.

## Сравнительная таблица

| Стратегия | Overhead | Адаптивность | Redis | Failover | Лучший сценарий |
|-----------|---------|-------------|-------|---------|----------------|
| Round-Robin | ~0 | Нет | Нет | Нет | Одинаковые реплики |
| Weighted | ~0 | Нет | Нет | Нет | Известные пропорции |
| Latency | ~1ms (Redis) | Да | Да | Нет | Переменная латентность |
| Least-Conn | ~0 | Да | Нет | Нет | Разная длительность запросов |
| Health-Aware | ~0 | Да | Нет | Да | Ненадёжные провайдеры |

Все стратегии автоматически фильтруют unhealthy через circuit breaker — health-aware стратегия дополнительно применяет round-robin только по здоровым.

## Результаты failover

Из нагрузочного теста (3 реплики, round-robin, circuit breaker threshold=5):

- **0% ошибок** при стабильной работе всех реплик
- **~16% ошибок** в момент отказа (переходный период до срабатывания CB)
- После CB: трафик перенаправляется, error rate стабилизируется
- Latency не деградирует при работе на меньшем числе реплик (p50 стабилен ~25ms)

## Рекомендации

Для production LLM gateway:

1. **Default** — `round-robin` с circuit breaker (покрывает 80% случаев)
2. **Мульти-провайдер** (OpenAI + Anthropic) — `latency` для автоматического выбора быстрейшего
3. **Стриминг** — `least-connections` для равномерной нагрузки при длинных streams
4. **Высокая доступность** — `health-aware` + failover + TTFT timeout
